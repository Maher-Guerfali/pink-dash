ğŸ”‘ KEY REACT CONCEPTS USED
================================================================================

1. COMPONENTS
   What: Reusable pieces of UI
   Used here: App.tsx, PatientList.tsx, PatientDetail.tsx
   Think of it as: A building made of LEGO blocks

   Example:
   - App.tsx is the main building
   - PatientList.tsx is a wall
   - PatientDetail.tsx is another wall
   - Each can be replaced or reused

2. HOOKS (Functions that let you use state and effects)

   a) useState
      What: Remembers data for your component
      Used here: 
      - searchTerm (what user typed in search box)
      - patients (list of patients from server)
      - loading (is data loading?)
      - error (did something go wrong?)
      
      Simple analogy:
      - Like variables in regular JS BUT
      - When they change, React AUTOMATICALLY re-draws the page

      Example from PatientList.tsx:
      const [patients, setPatients] = useState<Patient[]>([]);
      
      Breakdown:
      - patients = current value (starts empty [])
      - setPatients = function to change it
      - useState = React hook that makes this work
      
      When you call: setPatients(newList)
      React automatically updates the screen!

   b) useEffect
      What: Runs code at specific times
      Used here:
      - Fetch patients when component loads
      - Re-filter patients when search term changes
      
      Simple analogy:
      - Like event listeners in vanilla JS
      - "When X happens, do Y"
      
      Example from PatientList.tsx:
      useEffect(() => {
        fetchPatients();
      }, [refreshKey, retrieveCount]);
      
      Breakdown:
      - First [] = dependencies (what to watch)
      - When refreshKey or retrieveCount changes, run fetchPatients()
      - This replaces addEventListener() from vanilla JS

   c) useNavigate
      What: Changes the current page/URL
      Used here:
      - Click patient â†’ go to detail page
      - Click back â†’ go to list page
      
      Example:
      const navigate = useNavigate();
      navigate('/patient/123'); // goes to detail page for patient 123

   d) useParams
      What: Gets values from the URL
      Used here: PatientDetail.tsx gets patient ID from URL
      
      Example:
      const { id } = useParams<{ id: string }>();
      If URL is: /patient/abc123
      Then: id = "abc123"

3. PROPS (Passing data to components)
   What: How parent components send data to child components
   
   Example:
   <PatientList searchTerm={searchTerm} refreshKey={refreshKey} />
   
   PatientList receives:
   - searchTerm (what user typed)
   - refreshKey (a number that tells it to refresh)

ğŸ“Š WHAT IS FHIR?
================================================================================

FHIR = Fast Healthcare Interoperability Resources

It's a STANDARD format for healthcare data (like JSON schema).

Patient resource has:
- resourceType: "Patient" (always this)
- id: Unique identifier
- name: Array of HumanName objects
  * family: Last name
  * given: Array of first/middle names
- birthDate: YYYY-MM-DD format
- gender: "male", "female", "other", "unknown"
- address: Array of Address objects
  * line: Street address (array)
  * city: City name
  * state: State
  * postalCode: ZIP code
  * country: Country

Bundle is the SERVER RESPONSE format:
- resourceType: "Bundle"
- entry: Array of patient records
- total: How many total exist on server

This standardization means:
- Healthcare systems can share data
- Developers know what format to expect
- Real healthcare apps use this

ğŸ§ª COMMON BUG SCENARIOS (What could go wrong)
================================================================================

1. DATA DOESN'T LOAD
   - Check: Is FHIR server down?
   - Check: Is internet connection working?
   - Check: Browser console for error messages
   - Fix: Error handling already in code (shows error message)

2. SEARCH NOT WORKING
   - This is CLIENT-SIDE filtering (all in JavaScript)
   - It filters the patients you already loaded
   - It's case-insensitive
   - If no patients match, nothing shows

3. PAGINATION WRONG
   - PatientList shows 10 per page (fixed)
   - But you can retrieve 10, 20, 30, or 50 total
   - If you retrieve 10, only 1 page exists
   - If you retrieve 50, max 5 pages

4. CLICKING PATIENT DOES NOTHING
   - Make sure URL is changing
   - Check PatientDetail is rendering
   - Check useParams is getting the ID
   - Check PatientService.getPatientById is being called


ğŸ’¡ INTERVIEW PREP QUESTIONS YOU MIGHT GET
================================================================================

Q1: "What's the difference between useState and useEffect?"
A: useState remembers data and updates the page when it changes.
   useEffect runs code at specific times (like when component loads
   or when a dependency changes). useState = state, useEffect = side effects.

Q2: "Why did you use TypeScript?"
A: TypeScript catches type errors before runtime. In fhirService.ts,
   if we passed wrong data type, TypeScript warns us immediately.
   In big apps, this prevents bugs.

Q3: "How does data flow from backend to frontend?"
A: API request â†’ Server response â†’ axios receives data â†’ 
   useState updates state â†’ React re-renders component.

Q4: "Why use React Router instead of page refreshes?"
A: Single Page App (SPA) = faster, no flashing, better UX.
   Router changes view without full page reload.
   Users feel instant navigation.

Q5: "How does search work?"
A: Client-side filtering. Gets all patients first, then filters
   in JavaScript using .filter() and .includes().
   Faster for small datasets. For huge datasets, you'd do
   server-side filtering (?name=John in URL query params).

Q6: "What's axios vs fetch?"
A: Both make HTTP requests. Axios is simpler syntax, has built-in
   error handling, automatic JSON parsing. Fetch is native browser API.
   Both work fine.

Q7: "What's FHIR?"
A: Standard format for healthcare data. Like HTML is standard for web,
   FHIR is standard for medical records. Ensures different healthcare
   systems can share data.

Q8: "How would you add filtering to this?"
A: Currently client-side only. For backend filtering:
   const response = await fhirClient.get('/Patient', {
     params: {
       name: 'John',
       gender: 'male'
     }
   });

Q9: "How would you add authentication?"
A: Use JWT tokens. Get token from login endpoint, store in localStorage,
   send in axios headers with every request.
   Example: fhirClient.defaults.headers.common['Authorization'] = 'Bearer ' + token

Q10: "What's the scalability issue?"
A: Retrieving 50 patients works fine. But retrieving 5000 would be slow.
    Solution: Implement pagination on backend (offset/limit parameters).
    Currently pagination is frontend-only.

Q11: "Why use CSS Modules instead of plain CSS?"
A: CSS Modules scope styles to components. PatientList.module.css styles
    only PatientList component. Prevents naming conflicts in big apps.
    Plain CSS can cause conflicts when class names clash.
